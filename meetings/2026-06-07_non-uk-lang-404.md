# Meeting: 404 when filtering in a non-uk/ru UI language
Date: 2026-06-07

## What we discussed
Bug report: opening the site in a non-Ukrainian language, selecting "all Italy" +
"medicine", briefly shows results then 404s. Console showed RSC payload fetch
failures for `/uk/m/...` cards and a 404 on the `medicine` document.

## Root cause
The UI offers ~9 languages (`en, uk, ru, it, pt, de, fr, tr, es`) via
`state.lang` / `useTranslation()`, but the Next.js SEO layer (`web/`) only has
routes for `uk` and `ru` (`lib/i18n.ts` `LANGS`). The route resolvers call
`isLang(p.lang)` and `notFound()` on anything else.

The search button built the URL from the UI lang: `/{lang}/{city}/{category}`.
With UI lang = `en` and "all Italy" (empty city), this produced `/en/medicine`
→ `isLang("en")` false → `notFound()` → 404. The brief "snap" is the client
transition rendering from seed before the server 404 lands. (Also explains the
`/uk/m/...` RSC prefetch fallbacks — cards default to `uk`, then the 404 doc
breaks the prefetch.)

## Options considered
- Add a `urlLang()` clamp (UI lang → uk/ru) and use it everywhere a URL is built — chosen; smallest, matches existing `MasterCard` pathname clamp.
- Expand SEO `LANGS` to all 9 — rejected; needs real translated SEO content, big scope.
- Make routes tolerant of unknown langs (redirect to default) — rejected; canonical/hreflang would get messy.

## Decisions made
- Added `urlLang(x): Lang` to `web/lib/i18n.ts` — `ru` stays `ru`, everything else → `DEFAULT_LANG` (`uk`).
- Applied it to every UI-lang-driven URL builder:
  - `spa/pages/Main.tsx` — search navigation segments
  - `spa/components/Modal.tsx` — modal `pushState` master URL (`/{lang}/m/...`)
  - `spa/components/Root.tsx` — footer privacy link
- `MasterCard.tsx` already clamps via pathname, left as-is.
- Typecheck clean.

## Open questions
- The `_rsc` "access control checks" CORS errors — likely a downstream symptom of the 404 doc, not a separate bug. Re-verify in prod after deploy.
